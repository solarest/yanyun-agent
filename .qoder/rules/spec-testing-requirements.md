# Spec 生成测试要求规范

## 类型
始终生效

## 描述
在生成 spec（规范/需求文档）时，必须包含功能测试和单元测试的代码说明，并确保后续能够支持反复验证与子回归。

## 规则内容

### Spec 中的测试要求

#### 1. 功能测试说明

每个 spec 必须包含以下功能测试说明：

```markdown
## 功能测试计划

### 测试场景 1: [功能名称] - 正常流程
- **前置条件**: [执行测试前需要满足的条件]
- **测试步骤**:
  1. [步骤 1]
  2. [步骤 2]
  3. [步骤 3]
- **预期结果**: [期望的输出或行为]
- **验收标准**: [具体的判断标准]

### 测试场景 2: [功能名称] - 异常流程
- **前置条件**: [执行测试前需要满足的条件]
- **测试步骤**:
  1. [步骤 1]
  2. [步骤 2]
- **预期结果**: [期望的错误处理行为]
- **验收标准**: [具体的判断标准]

### 测试场景 3: [功能名称] - 边界条件
- **前置条件**: [执行测试前需要满足的条件]
- **测试步骤**:
  1. [步骤 1]
- **预期结果**: [期望的边界处理行为]
- **验收标准**: [具体的判断标准]
```

#### 2. 单元测试说明

每个 spec 必须包含以下单元测试说明：

```markdown
## 单元测试计划

### 测试模块: [模块名称]

#### 测试用例 1: [方法/函数名]_[场景]_[预期结果]
- **测试目标**: [验证的具体行为]
- **输入参数**: 
  - 参数1: [值]
  - 参数2: [值]
- **Mock 依赖**: [需要模拟的外部依赖]
- **预期行为**: 
  - 返回值: [期望的返回值]
  - 方法调用: [期望调用的方法]
  - 状态变更: [期望的状态变化]
- **测试代码框架**:
  ```python
  def test_[method]_[scenario]_[expected]():

      ...
      
      # Act
      ...
      
      # Assert
      ...
  ```

#### 测试用例 2: [方法/函数名]_[场景]_[预期结果]
...
```

### 反复验证与子回归支持

#### 1. 测试可重复性

所有在 spec 中定义的测试必须满足：

- **环境无关性**: 测试不应依赖特定环境配置
- **数据隔离**: 每个测试使用独立的测试数据
- **状态清理**: 测试前后进行必要的状态清理
- **确定性**: 相同输入总是产生相同输出

```python
# 正确 - 可重复的测试
def test_create_user_with_valid_data():
    # Arrange - 使用独立数据
    test_repo = InMemoryUserRepository()  # 每次创建新实例
    test_data = UserCreateDTO(name="Test", email="test@example.com")
    
    # Act
    user = test_repo.save(test_data)
    
    # Assert
    assert user.email == "test@example.com"
    # 不依赖外部状态或全局变量

# 错误 - 不可重复的测试
def test_create_user():
    # 使用全局状态，测试之间会相互影响
    global_user_repo.save(UserCreateDTO(...))
    assert global_user_repo.count() > 0
```

#### 2. 子回归测试策略

在 spec 中定义功能变更时，必须包含回归测试计划：

```markdown
## 回归测试计划

### 受影响的现有功能
- [ ] 功能 A: [描述影响范围]
- [ ] 功能 B: [描述影响范围]
- [ ] 功能 C: [描述影响范围]

### 回归测试用例
- **用例 1**: [原有功能] - 确保新变更不影响原有行为
- **用例 2**: [集成点] - 确保与其他模块的集成正常
- **用例 3**: [边界情况] - 确保边界条件仍被正确处理

### 自动化验证
```bash
# 运行相关测试套件
uv run pytest tests/ -k "affected_module"
uv run pytest tests/integration/
npm test -- --testPathPattern=affected_component
```
```

#### 3. 测试执行与验证流程

spec 中应包含完整的测试执行流程：

```markdown
## 测试执行流程

### 开发阶段验证
1. 编写单元测试
2. 运行单元测试: `uv run pytest tests/unit/`
3. 编写集成测试
4. 运行集成测试: `uv run pytest tests/integration/`
5. 运行完整测试套件: `uv run pytest tests/`

### 代码审查验证
1. 检查测试覆盖率: `uv run pytest tests/ --cov=src`
2. 验证测试独立性: `uv run pytest tests/ --forked`
3. 确认回归测试通过

### CI/CD 验证
1. 自动化运行所有测试
2. 生成覆盖率报告
3. 阻塞未通过测试的合并请求
```

### 实施检查清单

创建 spec 时，确保：

- [ ] 包含完整的功能测试场景（正常、异常、边界）
- [ ] 包含详细的单元测试用例
- [ ] 定义了回归测试范围
- [ ] 提供了测试执行命令
- [ ] 明确了验收标准
- [ ] 包含反复验证的流程说明
- [ ] 定义了测试维护责任
- [ ] 考虑了测试数据的隔离性
- [ ] 确保测试的确定性
- [ ] 包含了自动化验证方案

### 最佳实践

#### DO ✅

1. **测试先行**
   - 在实现前定义测试
   - 用测试验证需求完整性
   - 通过测试明确验收标准

2. **分层测试**
   - 单元测试验证逻辑
   - 集成测试验证交互
   - 功能测试验证业务

3. **持续验证**
   - 每次变更运行相关测试
   - 定期运行完整回归测试
   - 维护测试基线

#### DON'T ❌

1. **避免测试遗漏**
   - 不要只测试正常流程
   - 不要忽略异常场景
   - 不要忘记边界条件

2. **避免测试脆弱**
   - 不要依赖外部状态
   - 不要使用不稳定的数据
   - 不要创建相互依赖的测试

3. **避免测试过时**
   - 不要忽略失败的测试
   - 不要跳过回归测试
   - 不要降低测试标准

### 工具和自动化

#### 测试执行脚本

```bash
#!/bin/bash
# run_tests.sh - 运行所有相关测试

echo "=== 运行单元测试 ==="
uv run pytest tests/unit/ -v

echo "=== 运行集成测试 ==="
uv run pytest tests/integration/ -v

echo "=== 运行功能测试 ==="
uv run pytest tests/functional/ -v

echo "=== 检查覆盖率 ==="
uv run pytest tests/ --cov=src --cov-report=term-missing

echo "=== 运行前端测试 ==="
cd frontend && npm test

echo "=== 测试完成 ==="
```

#### 回归测试脚本

```bash
#!/bin/bash
# run_regression_tests.sh - 运行回归测试

# 获取变更的文件
CHANGED_FILES=$(git diff --name-only main...HEAD)

# 确定需要运行的测试
echo "=== 检测变更文件 ==="
echo "$CHANGED_FILES"

echo "=== 运行受影响测试 ==="
# 根据变更文件运行相关测试
uv run pytest tests/ -k "affected_by_changes"

echo "=== 回归测试完成 ==="
```

### 度量与改进

#### 测试健康度指标

- **测试覆盖率**: 代码覆盖率 ≥ 80%
- **测试通过率**: 100% 的测试必须通过
- **测试执行时间**: 完整测试套件 < 10 分钟
- **测试稳定性**: 无随机失败的测试
- **回归测试覆盖率**: 关键路径 100% 覆盖

#### 持续改进

1. **定期回顾**
   - 每月回顾测试覆盖率
   - 分析失败测试原因
   - 优化慢测试

2. **测试债务管理**
   - 记录缺失的测试
   - 优先级排序
   - 定期清理

3. **流程优化**
   - 收集反馈
   - 改进测试策略
   - 自动化重复工作
