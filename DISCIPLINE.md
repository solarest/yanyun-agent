## 代码规范

### 通用规则

1. **命名规范**
   - 使用有意义的、描述性的名称
   - 避免缩写和单字母变量（除循环计数器外）
   - 命名应反映业务领域概念

2. **注释规范**
   - 代码应自解释，注释用于说明"为什么"而不是"做什么"
   - 所有公共接口必须有文档注释
   - 复杂业务逻辑必须添加注释说明

3. **DRY 原则**
   - 避免重复代码，提取公共逻辑
   - 遵循单一职责原则（SRP）

### Python 后端规范

1. **代码风格**
   - 使用 `ruff` 进行代码格式化和检查
   - 遵循 PEP 8 规范
   - 行长度限制：100 字符

2. **类型注解**
   - 所有函数必须有类型注解
   - 使用 `mypy` 进行类型检查
   - 避免使用 `Any` 类型

3. **DDD 架构约束**
   - 领域层不能依赖其他层
   - 应用层只能依赖领域层接口
   - 基础设施层实现领域层接口
   - 表现层通过依赖注入使用应用层

4. **错误处理**
   - 使用自定义异常类
   - 不要在领域层捕获异常
   - 在表现层统一处理异常并返回适当响应

5. **依赖管理**
   - 使用 `uv` 管理依赖
   - 所有依赖必须在 `pyproject.toml` 中声明
   - 使用 `uv sync` 安装依赖

### TypeScript 前端规范

1. **代码风格**
   - 使用 ESLint 和 Prettier
   - 严格模式（strict: true）
   - 禁止使用 `any` 类型

2. **组件规范**
   - 使用函数组件和 Hooks
   - 组件文件使用 PascalCase 命名
   - 每个组件单独一个文件

3. **状态管理**
   - 遵循领域层定义的类型
   - API 调用必须在 infrastructure 层实现
   - 业务逻辑在 application 层处理

## Git 工作流

### 基本原则

1. **保持提交原子性**
   - 每个提交只做一件事
   - 相关改动放在同一个提交中

2. **频繁提交**
   - 完成一个功能点就提交
   - 不要累积大量改动

3. **推送前检查**
   - 运行测试确保通过
   - 检查代码格式
   - 确保类型检查通过

## 提交规范

### 提交消息格式

使用 Conventional Commits 规范：

```
<type>(<scope>): <description>

[optional body]

[optional footer(s)]
```

### Type 类型

- `feat`: 新功能
- `fix`: 修复 bug
- `docs`: 文档变更
- `style`: 代码格式（不影响功能）
- `refactor`: 重构
- `perf`: 性能优化
- `test`: 测试相关
- `chore`: 构建过程或辅助工具变动
- `ci`: CI/CD 配置变更

### 示例

```
feat(auth): add user login endpoint
fix(api): handle null pointer in ping service
docs(readme): update installation instructions
refactor(domain): extract base entity class
```

### 规则

- 使用中文或英文均可，但整个项目应保持一致
- 描述使用祈使句（"add" 而不是 "added"）
- 首字母不要大写
- 结尾不要加句号

## 分支管理

### 分支命名

- `main` - 主分支，始终保持可部署状态
- `develop` - 开发分支
- `feature/<name>` - 功能分支
- `bugfix/<name>` - 修复分支
- `hotfix/<name>` - 紧急修复分支
- `release/<version>` - 发布分支

### 分支策略

1. 从 `develop` 创建功能分支
2. 在功能分支上开发
3. 完成后创建 Pull Request 到 `develop`
4. 代码审查通过后合并
5. 定期从 `develop` 创建发布分支进行测试
6. 发布验证后合并到 `main` 并打 tag

## 代码审查

### 审查清单

提交 PR 前自查：

- [ ] 代码遵循项目规范
- [ ] 添加了必要的测试
- [ ] 所有测试通过
- [ ] 代码格式化完成
- [ ] 类型检查通过
- [ ] 文档已更新
- [ ] 没有遗留的 debug 代码
- [ ] 提交消息清晰准确

### 审查要点

审查他人代码时关注：

1. **正确性**
   - 逻辑是否正确
   - 边界条件是否处理

2. **可读性**
   - 命名是否清晰
   - 代码结构是否合理

3. **性能**
   - 是否有明显的性能问题
   - 是否有不必要的计算

4. **安全性**
   - 输入验证是否充分
   - 是否有安全隐患

5. **DDD 合规性**
   - 是否遵循分层架构
   - 依赖关系是否正确

### 响应时间

- PR 应在提交后 24 小时内得到审查
- 紧急修复优先处理

## 测试规范

### 测试层级

1. **单元测试**
   - 测试单个函数/方法
   - 覆盖核心业务逻辑
   - 位于 `tests/unit/`

2. **集成测试**
   - 测试模块间交互
   - 测试 API 端点
   - 位于 `tests/integration/`

3. **端到端测试**（前端）
   - 测试完整用户流程
   - 位于 `tests/e2e/`

### 测试规则

1. **命名规范**
   - 测试函数名应描述测试场景
   - 格式：`test_<function>_<scenario>_<expected>`

2. **覆盖率**
   - 核心业务逻辑覆盖率 ≥ 90%
   - 工具函数覆盖率 ≥ 80%

3. **独立性**
   - 测试之间不能相互依赖
   - 每个测试可以独立运行
   - 使用 Mock 隔离外部依赖

4. **AAA 模式**
   - Arrange（准备）- 设置测试数据
   - Act（执行）- 调用被测函数
   - Assert（断言）- 验证结果

### 运行测试

```bash
# 后端
cd backend
uv run pytest tests/
uv run pytest tests/ --cov=src  # 带覆盖率

# 前端
cd frontend
npm test
```

## 文档规范

### 文档类型

1. **代码文档**
   - 函数和类的 docstring
   - 复杂逻辑的行内注释

2. **API 文档**
   - 使用 FastAPI 自动生成（/docs）
   - 保持请求/响应示例更新

3. **项目文档**
   - README.md - 项目概述
   - DISCIPLINE.md - 开发纪律
   - CHANGELOG.md - 变更日志

### 文档维护

- 代码变更时同步更新文档
- API 变更时更新 API 文档
- 新功能添加时更新 README

## 安全规范

### 代码安全

1. **敏感信息**
   - 绝不提交密码、密钥、Token
   - 使用环境变量或配置文件
   - 将 `.env` 加入 `.gitignore`

2. **输入验证**
   - 所有外部输入必须验证
   - 使用 Pydantic 进行数据验证
   - 防止 SQL 注入、XSS 等攻击

3. **依赖安全**
   - 定期更新依赖
   - 使用 `uv` 锁定依赖版本
   - 关注安全公告

### 数据安全

- 敏感数据必须加密存储
- 遵循最小权限原则
- 记录安全相关的操作日志

## 开发流程

### 新功能开发

1. **需求分析**
   - 理解业务需求
   - 识别领域概念
   - 设计领域模型

2. **开发步骤**
   ```
   a. 在领域层定义 Entity 和 Repository 接口
   b. 在应用层实现 Use Case
   c. 在基础设施层实现 Repository
   d. 在表现层添加 API 端点
   e. 在前端添加对应功能
   f. 编写测试
   g. 更新文档
   ```

3. **验证**
   - 运行所有测试
   - 手动测试 API
   - 代码审查

### Bug 修复流程

1. 复现问题
2. 编写失败的测试用例
3. 修复代码
4. 确保测试通过
5. 回归测试
6. 提交修复

### 日常开发检查

每次提交前运行：

```bash
# 后端
cd backend
uv run ruff check src/
uv run ruff format src/
uv run mypy src/
uv run pytest tests/

# 前端
cd frontend
npm run type-check
npm run build
```
